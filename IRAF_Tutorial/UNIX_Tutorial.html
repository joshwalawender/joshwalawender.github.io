UNIX/Linux Tutorial

 

The Command Line


There are many “flavors” of the UNIX style operating system including all of the Linux variants as well as Mac OS X.  While each has its own graphical interface and tools which constantly change, in each case you can usually get to a “command line interface” in which you can interact with the operating system (OS) via a series of typed commands and responses.


Terminal


Each OS will have a program for interacting with the command line, in man cases this program is simply called terminal.  Another common program is called xterm.  Once you start that program you should be presented with a window which looks vaguely like the screenshot below (which is Terminal from Mac OS X).  You have some basic information on the top line which was printed when the program started (this may or may not be there depending on the individual program).  The important part is the second line which contains the “prompt”.


The prompt prints out some text (in this case “MacBook Pro” the name of the computer in brackets, then a ~ which represents the path (to be discussed below), then a > which indicates the end of the prompt and that the program is awaiting input.




If I type a command, the program will respond (if the command warrants a response).  For example I can type the command pwd:




The response is the current working directory.  Any time we are working with the command line we are doing everything at a particular location in the filesystem.  In this case, we are in the home directory of the user joshw.  The home directory for each user is the starting location for almost everything we do.  Because of this, it sometime gets a special notation.  The home directory is represented by the ~ symbol which is why we saw that symbol in the prompt.  In this case, the prompt is set to always display the current working directory.  They can be customized to display all sorts of information, so your prompt may differ.  When I typed the pwd command, I got to see two different representations of the home directory location.


We can also change the directory we are in with the cd command.




In this case I changed to a directory at the location /Volumes/Data/joshw.  That location is now seen in my prompt (your prompt may be different).  I can list the contents of that directory with the ls command.




In this example, I have used ls twice.  The first time it showed me the five folders in the current directory.  The second time, I invoked ls with an option flag, that’s the -l at the end of the command.  This told ls to show me the information in long format with one item per line, but with additional information about each item.  There are usually many options for the most commonly used commands, the best way to find out about them is through their manual page which can be seen by typing man ls (or man <command name>) at the prompt.  This will show the information in the terminal, however, I often find it easier to look at online versions of the information, so googling the command is often effective.


Now I can change back to somewhere within my home directory (I left my home directory when I changed to the /Volumes/Data/joshw directory).  Rather than using the full path to my home directory, I will use the ~ shortcut mentioned above:




This takes me to the Documents folder within my home directory.


Up to this point, we have used the “full path names” for the directories we wanted (sometimes using the ~ shortcut).  We can also move relative to our current directory.  If I wanted to move to the home directory from here, I could use the full path (which would be easy with the shortcut), but I could also use a symbol which means “one directory up”, that would look like this:  cd ..




Now I am in my home directory.  If I wanted to move into a particular directory housed within my current directory I would use its relative path.  This leaves off that initial / character which represents the lowest directory on the computer.  To go back in to Documents, I simply type:  cd Documents

Now you have the basics for navigating from location to location on the command line and listing directory contents.




Another useful command is more.  This program simply displays the contents of a text file on the screen and allows you to page through the contents with the spacebar.  The less command is similar, but allows you to move through the text file one line at a time with the up and down arrow keys.


To get to a full text editor, you can invoke that program name.  One common one is vi.  This program is very versatile and runs within the terminal window.


Often it is desirable to have a program you invoke at the command line open up a new window.  For example, you can get another terminal by typing the terminal command, for example, to get another xterm, just type xterm.  There is a problem with this however, the command line is designed with the idea that this terminal does one thing at a time.  You give a command, then it executes the command.  When the command is finished, you get a new prompt to enter the next command.  If you invoke a new xterm, your current terminal is now waiting for that program to finish and so is not responsive (you don’t get your next prompt).  The way around this is with the & symbol.  This tells the program to “run in the background” and not wait for this command to be finished before moving on.  Thus typing xterm & at the prompt will bring up a new xterm without using up the old one.


If your text editor program brings up its own window, you will probably want to use this.  For example, one of my preferred text editors is emacs.  To invoke it, I would usually type emacs &
